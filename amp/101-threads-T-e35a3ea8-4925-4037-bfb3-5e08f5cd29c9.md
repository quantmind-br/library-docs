## 2. Simplifying `VideoDetail`: Extract the layout and a couple of subcomponents

### 2.1 Extract the fixed 3-column shell

Right now `VideoDetail` is doing:

- Layout shell concerns (positioning left/right sidebars, padding for center).
- Page logic (start time, autoplay, scroll, playlist nav).
- Main content rendering (player, metadata, tabs, CTA, transcript).

The layout shell noise obscures the interesting logic. Extract to a reusable component; this can live in `components/layout/fixed-sidebar-layout.tsx` or next to the page as `video-detail-layout.tsx`.

## 3. Hooks / logic to extract or simplify

### 3.1 Derived values should stay derived, not in state

In `VideoDetail`:

```

    !!video.transcript &&
      !!video.transcript.json &&
      !!video.transcript.json.url

```

You never change `video`. And `hasTranscript` is a pure derivation from `video`.

**Refactor:**


Result: fewer hooks, fewer moving parts; intent is obvious.

### 3.2 Extract playlist navigation into a small hook

Right now:

```
const currentVideoIndex =
  playlist?.videos.findIndex((v) => v.id === video.id) ?? -1;
const hasPreviousVideo = playlist && currentVideoIndex > 0;
const hasNextVideo = playlist &&
  currentVideoIndex >= 0 &&
  currentVideoIndex < playlist.videos.length - 1;

const previousVideo = hasPreviousVideo
  ? playlist.videos[currentVideoIndex - 1]
  : null;
const nextVideo = hasNextVideo
  ? playlist.videos[currentVideoIndex + 1]
  : null;
```

Thatâ€™s good logic, but it clutters `VideoDetail`.

**Hook:**

```
// hooks/use-playlist-navigation.ts
import type { Playlist } from "@boldvideo/bold-js";

export function usePlaylistNavigation(playlist: Playlist | undefined, videoId: string) {
  if (!playlist) {
    return {
      currentIndex: -1,
      hasPrevious: false,
      hasNext: false,
      previous: null,
      next: null,
    };
  }

  const currentIndex = playlist.videos.findIndex((v) => v.id === videoId);
  const hasPrevious = currentIndex > 0;
  const hasNext = currentIndex >= 0 && currentIndex < playlist.videos.length - 1;

  return {
    currentIndex,
    hasPrevious,
    hasNext,
    previous: hasPrevious ? playlist.videos[currentIndex - 1] : null,
    next: hasNext ? playlist.videos[currentIndex + 1] : null,
  };
}
```

Use it in `VideoDetail`:

```
const {
  currentIndex: currentVideoIndex,
  hasPrevious: hasPreviousVideo,
  hasNext: hasNextVideo,
  previous: previousVideo,
  next: nextVideo,
} = usePlaylistNavigation(playlist, video.id);
```

This makes `handleVideoEnded` and the mobile controls read much cleaner.

### 3.3 Extract â€śscroll visibilityâ€ť into a hook (and fix the logic)

Current code:

```
const [isOutOfView, setIsOutOfView] = useState<boolean>(false);
const prevScrollY = useRef(0);

const handleScroll = useCallback(() => {
  const currentScrollY = window.scrollY;
  if (currentScrollY > window.innerHeight * 0.7 && !isOutOfView) {
    setIsOutOfView(false);
  }
  if (currentScrollY < window.innerHeight * 0.7 && isOutOfView) {
    setIsOutOfView(false);
  }
  prevScrollY.current = currentScrollY;
}, [isOutOfView]);
```

This currently never sets `true` and the two branches call `setIsOutOfView(false)` in both cases â€“ clearly a smell/bug.

**Hook:**

```
// hooks/use-scroll-out-of-view.ts
import { useEffect, useState } from "react";

export function useScrollOutOfView(thresholdRatio: number = 0.7) {
  const [isOutOfView, setIsOutOfView] = useState(false);

  useEffect(() => {
    const handleScroll = () => {
      const currentScrollY = window.scrollY;
      const threshold = window.innerHeight * thresholdRatio;
      setIsOutOfView(currentScrollY > threshold);
    };

    window.addEventListener("scroll", handleScroll, { passive: true });
    handleScroll(); // initialize
    return () => window.removeEventListener("scroll", handleScroll);
  }, [thresholdRatio]);

  return isOutOfView;
}
```

Then in `VideoDetail`:

```
const isOutOfView = useScrollOutOfView(0.7);
```

Now the `Player` prop is a simple, correct boolean, and the scroll behavior is encapsulated and testable.

## 4. Patterns that smell (and suggested cleanup)

1. **â€śLegacyâ€ť props in `VideoContentTabs` that arenâ€™t honored.**

## 5. Recommended refactor plan (simple path)

1. **Rename the right sidebar component:**

This path keeps your existing concepts and dependencies, but makes each file much more readable and narration-like.

## 6. Rationale and trade-offs

- **Why this is enough now:**
  
  - Your current architecture is fundamentally sound: a fixed 3-column app shell with stateful sidebars.
  - The pain today is *readability*, not capability. Extracting layout and a couple of hooks/components directly addresses the â€ścode singingâ€ť goal without over-engineering.
- **Why not a deeper layout system?**
  
  - You could build a generic â€śAppShellâ€ť for the whole app, but thatâ€™s a bigger design decision and not required just to make this specific page clear.
  - YAGNI: refactor this page cleanly first; factor out to a larger shared layout only if multiple pages start to need the same pattern.

## 7. Risks and guardrails

- **Risk: subtle CSS / spacing regressions** when extracting `FixedSidebarLayout`.
  
  - Guardrail: keep the exact padding calculations and classes; write a visual regression note (or snapshot) for desktop + mobile with/without playlist.
- **Risk: behavior around `isOutOfView` changes.**
  
  - Guardrail: confirm what the `Player` does with `isOutOfView` (e.g., mini-player behavior). Test scroll behavior manually after moving to `useScrollOutOfView`.
- **Risk: localStorage / sessionStorage interactions in sidebars.**
  
  - Guardrail: test a) first load, b) reload, c) resize across mobile/desktop, focusing on right sidebar tab persistence and open/closed state.
