Skills are registered and merged through a multi-layered system involving the `SkillService`, `SkillLoader`, and `MCPService`.

### Key Findings

- **Discovery & Loading**: Skills are discovered from `.agents/skills/` (workspace) and `~/.config/agents/skills/` (global). The `SkillLoader` parses `SKILL.md` for metadata and `mcp.json` for bundled MCP servers.
- **MCP Bundling**: Skills use `mcp.json` to define MCP servers. These are automatically started at startup but filtered via `includeTools` glob patterns to prevent context bloat.
- **Tool Registration**: The `MCPService` subscribes to the derived `skillMCPServers` stream and registers tools with the `ToolService` using `registerTool`. It handles incremental updates and deduplication.
- **Merging Strategy**: `SkillService` merges fresh scans with cached results, retaining existing skills unless they explicitly fail to load or are deleted, ensuring a stable experience during reloads.
- **Built-in Skills**: Built-in skills (in `core/src/skills/builtin/`) can unlock "deferred tools"—tools that are registered but hidden until the specific skill is activated in a thread.

Findings

- `SkillService` builds `skillMCPServers` as a plain record keyed by the server name from `mcp.json`, so later skills overwrite earlier ones (skills are sorted, so last name wins). That means only one `includeTools` set survives. See the construction in [core/src/skills/service.ts](file:///Users/nicolaygerold/code/amp/core/src/skills/service.ts#L335-L360).
- `MCPService` uses that map as the authoritative server list and keys connections by server name, so duplicates collide and only one MCP client is used. See the merge and connection map in [core/src/mcp/service.ts](file:///Users/nicolaygerold/code/amp/core/src/mcp/service.ts#L344-L522).
- Tool registration uses `includeTools` from the selected spec to filter tools; if the wrong spec wins, you only get that subset. See [core/src/mcp/service.ts](file:///Users/nicolaygerold/code/amp/core/src/mcp/service.ts#L760-L787).

Proposal (recommended)

- Merge duplicate skill MCP server entries by name when the underlying server spec is the same.
  
  - In [core/src/skills/service.ts](file:///Users/nicolaygerold/code/amp/core/src/skills/service.ts#L335-L360), detect duplicate `serverName`s across skills.
  - If the server config matches (same `command`/`url`/`headers`/etc.), union `includeTools` across skills so all tools are registered.
  - Track multiple owners: extend `SkillMCPServerSpec` to hold `_ampSkillNames?: string[]` (or similar) instead of a single `_ampSkillName`, and update any UI paths that display `skillName` (CLI/TUI MCP status). The MCP tool meta can stay hidden-from-default-listing; this change only affects the “via skill” label.
  - If two skills use the same server name but *different* specs, log a warning and auto-rename or skip one to avoid undefined behavior (see alternative below).

Alternative (more breaking)

- Enforce unique server names by prefixing with skill name (the comment already claims this but it is not implemented). That means `skill__${skillName}__${serverName}` when building `skillMCPServers` in [core/src/skills/service.ts](file:///Users/nicolaygerold/code/amp/core/src/skills/service.ts#L335-L360).
- This avoids collisions completely but changes tool names (`mcp__...` uses server name in [core/src/mcp/service.ts](file:///Users/nicolaygerold/code/amp/core/src/mcp/service.ts#L1160-L1208)), so existing skill instructions referencing tool names would need updates.

Why the merge option is safer

- It preserves existing tool names and avoids new MCP client instances.
- It solves the “last registration wins” issue by combining `includeTools`.
- It limits the surface area to `SkillMCPServerSpec` and UI labels, without changing tool identifiers.

Short answer: no — we should keep filtering at registration time, not during skill loading.

Why

- Tools are registered globally in `registerToolsWithToolService` and then hidden until the skill is activated. If we register all tools, we lose the ability to keep tool inventory small and deterministic per skill at the tool-service layer.
- The includeTools filter is intentionally applied before registering tools in [core/src/mcp/service.ts](file:///Users/nicolaygerold/code/amp/core/src/mcp/service.ts#L740-L807), which keeps the ToolService registry clean and avoids extra tool descriptions in memory.
- Skill loading should only control visibility/activation, not affect which tools are registered. That separation is baked into the current flow (skill → server specs → MCP connections → tool registration).

If we moved filtering to skill loading, we’d still need to register all tools first, then do a second filter when the skill activates. That defeats the purpose of `includeTools` and makes unrelated skills influence the global tool list.

The regression you’re seeing isn’t from “filtering location.” It’s from how we merged `includeTools` across skills in [core/src/skills/service.ts](file:///Users/nicolaygerold/code/amp/core/src/skills/service.ts#L330-L403). We merged the lists globally, so each skill now inherits the union. We should instead keep per-skill tool filters and apply them based on the activated skill when constructing tool visibility for the model.
